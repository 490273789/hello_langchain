# Milvus 学习笔记

## 1. Milvus 是什么

Milvus 是一个专门面向 **向量检索（Vector Search）** 的数据库，主要用于存储高维向量、建立向量索引，并执行相似度搜索。

如果把传统数据库理解为“擅长精确匹配和结构化查询”，那么 Milvus 更擅长解决下面这类问题：

- 找和某句话语义最接近的文本片段
- 找和一张图片特征最相似的其他图片
- 找和一个用户兴趣向量最接近的内容

在大模型应用中，Milvus 常用于：

- RAG（检索增强生成）
- 语义搜索
- 推荐系统召回
- 图像相似搜索
- 多模态检索

一句话理解：

> Milvus 是一个为 embedding 向量设计的数据库，核心能力是“快速找到最相似的向量”。

---

## 2. 为什么需要 Milvus

在做语义搜索时，通常会先用 embedding 模型把文本转成向量。例如：

```python
[0.12, -0.03, 0.88, ...]
```

如果不使用向量数据库，最直接的做法是：

1. 把所有文本都转成向量
2. 用户提问时，把问题也转成向量
3. 用问题向量和所有文本向量逐个比较距离
4. 选出最相似的前几个结果

这种方法在数据量小的时候可以工作，但当数据规模变大后会有几个问题：

- 检索速度慢
- 内存占用高
- 不方便扩展
- 难以支持过滤、分区、服务化部署

Milvus 的作用就是解决这些工程问题：

- 存储向量
- 建立索引
- 快速近似搜索
- 支持元数据过滤
- 支持大规模数据和分布式部署

## 3. Milvus 的核心概念

### 3.1 Vector（向量）

向量是 embedding 模型输出的数值表示。语义相近的内容，在向量空间中通常也更接近。

例如：

- “如何学习 Python”
- “Python 入门方法”

这两句话经过 embedding 后，得到的向量通常会比较接近。

### 3.2 Collection

Collection 类似传统数据库中的“表”。

例如你可以创建一个 `documents` collection，用来存储知识库中的文本片段。

### 3.3 Field

Field 类似表中的列。常见字段包括：

- `id`：主键
- `text`：原始文本
- `embedding`：向量字段
- `source`：来源
- `category`：分类
- `timestamp`：时间戳

### 3.4 Entity

Entity 表示一条记录，通常包含：

- 一段文本 chunk
- 对应的 embedding 向量
- 一些 metadata

### 3.5 Index

索引用于加速向量检索。因为高维向量直接逐条比较会很慢，所以 Milvus 会为向量字段建立专门的近似搜索索引。

### 3.6 Search

Search 是向量相似度搜索。输入一个 query vector，返回最相似的 top-k 个结果。

### 3.7 Filter / Query

除了按向量搜索，Milvus 还可以基于标量字段过滤，例如：

- 只搜索 `category = "LangChain"`
- 只搜索 `source = "manual"`
- 只搜索最近 30 天的数据

## 4. Milvus 在 RAG 中的作用

一个典型的 RAG 流程如下：

```text
原始文档
  -> 文档切分
  -> Embedding 向量化
  -> 存入 Milvus
  -> 建立索引

用户问题
  -> Embedding 向量化
  -> 在 Milvus 中做相似度搜索
  -> 取回最相关的文本片段
  -> 交给 LLM 生成答案
```

也可以画成更直观的流程：

```text
┌─────────────┐
│ 原始文档     │
└──────┬──────┘
	│
	v
┌─────────────┐
│ 文本切块     │
└──────┬──────┘
	│
	v
┌─────────────┐
│ Embedding   │
└──────┬──────┘
	│
	v
┌─────────────┐
│   Milvus    │
│ 向量存储检索 │
└──────┬──────┘
	│
	v
┌─────────────┐
│ 召回结果     │
└──────┬──────┘
	│
	v
┌─────────────┐
│     LLM     │
└─────────────┘
```

这里要注意分工：

- Embedding 模型负责把文本表示成向量
- Milvus 负责快速检索相似向量
- LLM 负责根据召回内容生成最终答案

Milvus 不负责“理解”问题，它负责“找出可能相关的内容”。

## 5. 向量相似度如何计算

Milvus 支持多种距离度量方式，最常见的有：

- `L2`：欧氏距离，越小越相似
- `IP`：内积，通常越大越相似
- `COSINE`：余弦相似度，通常越大越相似

在文本 embedding 场景里，`COSINE` 很常见。

余弦相似度公式为：

$$
	ext{cosine similarity} = \frac{A \cdot B}{\|A\|\|B\|}
$$

你可以这样理解：

- 两个向量方向越接近，相似度越高
- 值越接近 $1$，通常表示越相似

对于大多数文本语义检索任务，先关注 `COSINE` 就够了。

## 6. 常见索引类型

Milvus 的强项之一就是支持多种向量索引。初学时不需要把参数全部背下来，先理解不同索引的定位。

### 6.1 FLAT

`FLAT` 是最直接的精确检索方式。

特点：

- 结果最准确
- 检索速度相对慢
- 适合小规模数据或学习原理

### 6.2 IVF_FLAT

先把向量划分到多个桶里，搜索时只查部分桶。

特点：

- 检索更快
- 属于近似搜索
- 需要调节 `nlist`、`nprobe`

### 6.3 IVF_PQ

在 IVF 基础上做向量压缩。

特点：

- 更省内存
- 更适合更大规模数据
- 精度可能进一步下降

### 6.4 HNSW

基于图结构的近邻索引，是实际项目中常见的选择。

特点：

- 通常速度和召回效果都不错
- 查询性能优秀
- 索引构建成本和内存开销相对较高

### 6.5 DISKANN

适用于超大规模数据和磁盘友好场景。

特点：

- 适合海量向量检索
- 更偏工程优化与大规模部署场景

### 6.6 初学阶段怎么选

可以先这样记：

- 学习原理、小数据实验：`FLAT`
- 平衡性能和效果：`HNSW`
- 较大规模经典方案：`IVF_*`

## 7. 常见搜索参数

### 7.1 top_k

返回最相似的前几个结果。

例如：

- `top_k = 3`
- `top_k = 5`
- `top_k = 10`

在 RAG 中通常会取 3 到 10 条，然后交给 LLM 处理。

### 7.2 metric_type

表示使用什么距离计算方式，比如：

- `COSINE`
- `L2`
- `IP`

### 7.3 nlist 和 nprobe

这两个参数常见于 IVF 系列索引。

- `nlist`：分桶数量
- `nprobe`：查询时探测多少个桶

一般来说：

- `nprobe` 越大，召回率可能越高
- 但查询会更慢

### 7.4 HNSW 参数

常见参数有：

- `M`
- `efConstruction`
- `ef`

可以简单理解为：

- 控制图结构的复杂度
- 控制建图质量
- 控制查询时搜索范围

不要一开始就陷入调参，先把检索流程跑通更重要。

## 8. Milvus 中通常存什么数据

很多初学者会误以为 Milvus 只存向量。实际上，在真实项目中通常会一起存：

- 主键 id
- 向量字段
- 原始文本
- metadata

例如一条记录可能长这样：

```python
{
    "id": 1,
    "text": "Milvus is a vector database for similarity search.",
    "category": "database",
    "source": "docs",
    "embedding": [...]
}
```

这样做的好处是：

- 可以直接返回原始文本
- 可以结合 metadata 过滤
- 可以支持多租户、时间范围、分类检索等需求

## 9. Milvus 和传统数据库的区别

可以从查询目标来理解差异：

| 系统 | 擅长内容 |
|------|----------|
| MySQL / PostgreSQL | 结构化数据、精确过滤、事务 |
| Elasticsearch | 关键词检索、倒排索引、全文搜索 |
| Milvus | 向量相似搜索、近邻检索 |

真实项目中，它们往往是配合使用的，而不是相互替代：

- PostgreSQL 存业务数据
- Elasticsearch 做关键词搜索
- Milvus 做语义召回

---

## 10. Milvus 和 FAISS、Chroma 的区别

### 10.1 FAISS

FAISS 更像一个向量检索库，不是完整的数据库系统。

特点：

- 适合研究和本地实验
- 适合理解向量索引原理
- 工程化能力不如 Milvus 完整

### 10.2 Chroma

Chroma 更偏向轻量级、开发友好的向量数据库。

特点：

- 上手快
- 本地实验方便
- 适合原型和小规模项目

### 10.3 Milvus

Milvus 更偏生产级系统。

特点：

- 适合大规模向量检索
- 索引能力更强
- 更适合服务化和分布式部署

可以这样记：

- 学原理：FAISS
- 做 demo：Chroma
- 做工程系统：Milvus

## 11. Milvus 的典型应用场景

### 11.1 RAG 知识库问答

把文档切块后存入 Milvus，根据用户问题召回相关片段，再交给 LLM 回答。

### 11.2 语义搜索

用户输入一句自然语言，系统返回语义最接近的内容，而不是只看关键词是否完全匹配。

### 11.3 推荐系统召回

将用户和物品映射到同一向量空间，检索最接近用户兴趣的物品。

### 11.4 图像相似搜索

图片通过视觉模型编码成向量，Milvus 根据图像向量找最相似图片。

### 11.5 多模态检索

文本、图片、音频等都能转成向量后统一检索。

## 12. 学习 Milvus 时常见误区

### 12.1 只关注数据库，不关注 embedding 质量

向量数据库只能搜索“已有向量表达”中的相似度。如果 embedding 模型效果差，Milvus 也很难检索好。

### 12.2 chunk 切分不合理

- chunk 太大：主题混杂，召回不精准
- chunk 太小：上下文不完整，答案质量下降

### 12.3 把 ANN 当成精确搜索

Milvus 常用的是近似最近邻搜索（ANN），目标是“更快地找足够好的结果”，而不是绝对精确。

### 12.4 不做 metadata filter

如果所有数据都混在一起检索，噪声会很多。合理的分类、来源、时间过滤通常非常重要。

### 12.5 一开始就沉迷调参

初学阶段更重要的是：

- 跑通完整流程
- 看懂召回结果
- 分析检索质量

比起一上来调很多索引参数，这些更有价值。

## 13. 一个实用的心智模型

可以用下面这句话来记 Milvus：

> Milvus = 向量存储 + 向量索引 + 相似度搜索 + 条件过滤

在 RAG 系统中：

- LLM 负责生成
- Embedding 模型负责语义表示
- Milvus 负责召回相关内容

这个分工一定要分清。

## 14. Python 场景下的最小理解示例

下面是一个概念性示例，帮助理解 Milvus 在代码中的位置。

```python
# 伪代码：展示流程，不是完整可运行代码

documents = [
    "Milvus is a vector database.",
    "LangChain can integrate with vector stores.",
    "Embeddings convert text into vectors.",
]

# 1. 文本转向量
embeddings = embedding_model.embed_documents(documents)

# 2. 存入 Milvus
milvus.insert([
    {"id": 1, "text": documents[0], "vector": embeddings[0]},
    {"id": 2, "text": documents[1], "vector": embeddings[1]},
    {"id": 3, "text": documents[2], "vector": embeddings[2]},
])

# 3. 用户提问
query = "What is a vector database?"
query_vector = embedding_model.embed_query(query)

# 4. 相似度检索
results = milvus.search(query_vector, top_k=2)

# 5. 返回最相关文本
for item in results:
    print(item["text"])
```

这里最重要的不是 API，而是理解流程：

- 文本先向量化
- 再进入向量数据库
- 查询时同样先向量化
- 最后做相似度搜索

## 15. 如果结合 LangChain，重点应该学什么

如果你的目标是用 LangChain 学 Milvus，建议优先关注以下问题：

1. 文档如何切块
2. 选择什么 embedding 模型
3. collection 结构如何设计
4. 向量字段和 metadata 如何保存
5. 检索时如何加 filter
6. 是否需要 rerank
7. `top_k`、`metric_type`、索引类型如何选择

这些问题比单纯背 API 更重要。

---

## 16. 推荐学习顺序

建议按下面顺序学习：

1. 理解 embedding 是什么
2. 理解余弦相似度和 top-k 检索
3. 学会 collection、field、index、search 这些基本概念
4. 做一个最小 RAG 示例
5. 再研究 HNSW、IVF、过滤、混合检索和性能调优

---

## 17. 总结

Milvus 的定位可以总结为：

- 它不是传统关系型数据库
- 它不是只做关键词搜索的全文检索引擎
- 它是为高维向量相似搜索设计的数据库

对于大模型应用来说，Milvus 最重要的价值是：

- 帮助系统从大量向量中快速找出最相关的内容
- 为 RAG、语义搜索、推荐系统提供高效检索能力

学习时要抓住三个关键点：

- 向量从哪里来：Embedding
- 相似度如何算：Metric
- 如何快速检索：Index

只要这三件事理解清楚，Milvus 的整体脉络就基本建立起来了。
